# DouDou 平板端交互设计（reMarkable Paper Pro）

日期：2026-07-22（同日修订：补一期/二期能力划分、交互×教学活动映射，修正屏幕规格）

## 1. 背景与目标用户

DouDou 三件套：手机（听说）+ 服务器（记忆/教学策略）+ reMarkable Paper Pro（纸面交互）。

- **第一位真实用户是 3-4 岁孩子（涂鸦期，不识字）**。v1 把这个年龄段做扎实，5-6 岁（认字启蒙）与 6-7 岁（识字/写字）只做架构预留（见第 9 节）。
- **声音是一等公民**：平板无扬声器，语音由手机端承担。平板负责图像的输入与输出，手机负责语音层面；一个回合的反馈经常两端同时出。
- 现有原型 device/riddle：原始笔事件 → 停笔 2.8s → 整页 PNG → 视觉 LLM → 回复经 PingFangShiGuang.ttf 骨架化逐笔动画写回。本设计在此管线上演进。
- 屏幕：11.8 英寸 2160 × 1620 @229 PPI 竖屏，**Canvas Color 3.0 彩色电子纸**（E Ink Gallery 3 系）。彩色波形刷新慢且伴随闪烁，不适合高频回应；**v1 交互全部按灰度线稿设计**，彩色留作远期「特别时刻」表达选项（见第 11 节）。
- 服务器一期（已定稿）：平板零改动，riddle 走 OpenAI 兼容门面接入本地 Mac 服务器；/turn 结构化接口二期实施。本文档第 13 节划分了「一期能力内的交互」与「需要二期新渲染能力的交互」。

## 2. 设计原则

1. **纸的隐喻不破功**——无按钮、无菜单、无图标；笔是孩子在板上唯一的输入。
2. **双子通道**——纸面（平板）是看得见的想法，声音（手机）是听得懂的解说；同一回合同时落在两个通道。3-4 岁阶段语音承担意义传达，纸面承担画面与情感。
3. **孩子的作品神圣**——永不覆盖、永不淡化、永不「喝掉」孩子的墨迹（明确改掉 riddle 的吃墨水行为；淡出机制仅保留给「召回旧页」场景）。
4. **平板无教学状态**——活动、难度、分龄档位、长期记忆全在服务器；平板只懂纸和几何。
5. **慢即性格**——电子纸的慢与灰不是缺陷：逐笔动画就是「DouDou 在认真画画」的人格表达。

## 3. 核心交互循环（二期完整形态）

```
孩子画画 ──┬─ 停笔约 6 秒（服务器可配 5-8s）+ 本回合有新笔迹 ──┐
           └─ 手机端语音提问（随时，经服务器推 commit_now）────┤
                                                              ▼
        提交回合：整页 PNG + 本回合新增笔画（有序折线+压感+时间戳）+ 触发方式
                                                              ▼
        思考指示：页角一枚小墨点轻轻呼吸——不挡画面，孩子可继续画
                                                              ▼
        服务器返回 spoken_text + paper_cards[]
                                                              ▼
        手机播语音  ∥  平板在空白处逐笔画出卡片（避开孩子墨迹）
                                                              ▼
        回应留在纸上成为页面一部分，不淡出；页面快满时走换页流程
```

细则：

- **触发（双通道）**
  - 停笔触发：空闲阈值默认 6 秒（服务器按档位可配 5-8 秒，riddle 现值 2.8 秒对幼儿太急）；且本回合新墨迹 ≥ 2-3 笔才提交，防止误触小点触发。
  - 语音触发：家长/孩子对手机说话 → 服务器向平板推 `commit_now` → 平板立即封存当前页面为回合上下文。
  - 静默规则：DouDou 回应后不再计时，直到孩子画出新笔迹——避免自说自话循环。
- **思考期不锁页**：与 riddle 的 Drinking/Thinking 全屏状态机不同，思考中孩子可继续画。回应**落笔时**才解析布局，使用当时最新的墨迹分布。
- **回合并发**：同一时间只有一个回合在飞。回应书写中来了语音新回合 → 写完当前笔画即切换；停笔触发始终「最新覆盖旧等待」。
- **回应不淡出**：星星、简笔画留在纸上，与孩子的画共同构成这一页。

## 4. 卡片语言（paper_cards 词汇表 v1）

| 类型 | 内容 | 3-4 岁典型用途 | 渲染来源 |
|------|------|---------------|---------|
| `text` | 手写短句/单字 | 大字当图形、名字、数字 | 字体骨架化管线（现有） |
| `sketch` | 归一化坐标折线组 | 简笔画回应、补全、示范 | 服务器下发矢量 |
| `stamp` | 命名符号 + 数量 | 星星、红花、笑脸、对勾（奖励/庆祝） | **平板内置符号库** |
| `count` | 数量 n + 样式 | 数数点 ①②③（认知标注） | 平板生成 |
| `trace` | 描红模板（形状/汉字） | 描圆圈星星；5 岁后描汉字 | 字体/形状骨架淡墨渲染 |
| `page` | 整页家具 | 练习页、故事场景底图 | 组合以上元素铺整页 |

卡片通用字段：

- `place` 摆放意图（五种）：`near_new_ink`（贴着孩子刚画的）/ `near_anchor`（配 `anchor_norm`）/ `blank_area` / `margin` / `full_page`。
- `anchor_norm: [x, y]`（可选）：视觉模型看整页 PNG 时给出的归一化位置提示（「三个太阳大概在这里」）。
- `size`：S / M / L（相对页宽的建议尺寸，约 15% / 30% / 50% 页宽，具体常数实现期定）。
- `pace`：`normal` / `slow`（slow 用于希望孩子跟着看的示范，如描红前的慢速演示）。

简笔画传输格式决策：服务器下发**归一化坐标的有序折线组**，平板复用现有笔画动画引擎渲染。折线内容从哪来（精选素材库 / QuickDraw 检索 / 扩散模型生成后提线，见第 16 节）是服务器端决策，平板不关心。高频奖励符号（stamp）内置在平板，服务器只说名字和数量——保证最常用反馈零依赖、样式稳定。

## 5. 交互形式 × 教学活动映射（供课程设计引用）

每种交互形式适合承载的教学活动。语音列 = 手机端 `spoken_text` 在该活动中的分工（二期联动前由家长陪读代替）。

| 交互形式 | 适合的教学活动 | 适龄 | 语音分工 |
|----------|---------------|------|---------|
| `text` 大字当图形 | 字形熟悉（孩子画山→旁边写大大的「山」）、数字认知、认自己的名字 | 3-4 看形混脸熟；5+ 认读 | 读出这个字，把字音和字形钉在一起 |
| `text` 短句 | 儿歌/古诗跟读、简单问答、看图说话的文字落点 | 5+ | 领读、逐句解释 |
| `sketch` 简笔画示范 | 跟画课（「先画一个圆，再加两只耳朵」分步示范）、观察力游戏（豆豆画只猫，问猫有什么）、想象补全（豆豆画一半，孩子补另一半） | 3-4 起全龄 | 逐步解说每一笔在画什么 |
| `stamp` 奖励符号 | 即时奖励、任务完成庆祝、集星星长线激励（服务器记账，跨页累计） | 全龄 | 表扬语，点明因为什么被奖励 |
| `count` 数数标记 | 点数对应（画了几个苹果就点几个点）、比多少、简单加减（5+） | 3-4 数数；5+ 运算 | 带着孩子一个一个数出声 |
| `trace` 形状描红 | 运笔控制、手眼协调、基础图形认知（圆/方/三角/星/心/波浪线） | 3-4 | 节奏鼓励（「慢慢描，转弯啦」） |
| `trace` 汉字描红 + 田字格 | 笔顺、书写规范；配合回合上传的有序笔画可做笔顺点评 | 5-7 | 笔顺口诀、逐笔提示 |
| `page` 练习页 | 集中练习（一页四个描红格）、连线题、找不同（线稿版） | 5+ 为主 | 出题与讲解 |
| `page` 故事场景 | 豆豆铺一个场景底图（雨天的街道），孩子往里添画（给行人画伞）——共创式讲故事 | 3-4 起全龄 | 讲故事、引导「接下来画什么」 |
| 召回浮现（旧页淡墨重写） | 复习回顾（「还记得上周画的猫吗」）、作品集重温、成长对比 | 全龄 | 回忆引导 |

课程设计注意两条硬约束：每回合纸面最多 3 张卡片（第 14.3 节）；一切需要即时反馈或颜色语义的环节走语音/实物，不上这块屏（第 11 节）。

## 6. 布局器（平板端新组件，二期）

平板维护一张**墨迹覆盖图**（低分辨率网格，约 64×85 格，标记每格有无墨）。收到卡片后：

1. 按 `place` 意图确定搜索起点：新墨迹外接框旁 / `anchor_norm` 换算的页面坐标附近 / 全页扫描空白。
2. 螺旋式向外寻找能容纳 `size` 的空白矩形，与已有墨迹保持约 40px 呼吸边距。
3. 放不下 → 逐档缩小（L→M→S）；仍不行 → 本回合只渲染最重要的第一张卡片，并在下一次回合请求的 `ink_coverage` 中向服务器反映页面已满（服务器据此发起换页）。

`page` 卡是唯一例外：整页家具由服务器用 `rect_norm` 显式排版——此时是新白纸，没有竞态。

## 7. 手势体系（儿童防护档）

| 输入 | 行为 | 相对现状的变化 |
|------|------|---------------|
| 写画后停笔 ~6s | 提交回合 | 从 2.8s 加长；需有新墨迹 |
| 笔倒转 | 橡皮擦 | 保留 |
| 电源键 | 睡眠/唤醒（页面原样保留） | 保留 |
| 五指**长按 ≥3 秒** | 退出到原生系统 | 从「点按」改为长按，防手掌误触 |
| 画大「?」 | 儿童档**停用** | 求助走语音；识字档可由服务器经 `set_profile` 重新启用 |
| 家长手机指令 | 退出/重启/换页/切活动 | 新增（经服务器控制通道，二期） |

触摸（手指）在儿童档只保留五指长按一个入口，其余触摸一律忽略——手掌、胳膊肘趴上去无任何效果。误退出的后果（暴露 reMarkable 原生系统）由「长按 3 秒」的时间门槛防住：幼儿拍屏是瞬时接触。

其中「空闲 2.8s→6s」与「五指点按→长按」是两行常量级改动，不依赖 /turn——可作为一期后的首个设备微改动提前落地（见第 13.3 节）。

## 8. 页面与记忆

- 页面是一条**无限纸卷**：当前页快满时进入换页流程——DouDou 语音建议（手机）+ 纸面小动画（平板）→ 自动翻新页。家长/孩子也可随时语音「换一页」。孩子在板上没有可误触的换页手势。
- 旧页**一律自动存档**，不存在丢失。存档内容：有序笔画（含时间戳）+ 页面 PNG + 转写 + DouDou 回应卡片 + `memory_tags`。
- **服务器是记忆主库**（二期）：每页封存时平板上传完整存档；平板本地 `/home/root/riddle-data` 降级为缓存。手机端与管理后台由此能看到作品流。（一期：记忆仍全在平板，服务器门面已保留 riddle 记忆协议，召回照常。）
- **召回旧页**：保留 riddle 的浮现机制（旧页以淡墨逐笔重写，笔触任意处即消散回今天的页面）。触发入口对 3-4 岁改为语音（「豆豆，给我看昨天画的猫」→ 服务器推 `recall`，二期）；手写「给我看…」触发留给识字档（一期唯一入口）。

## 9. 分龄演进（平板视角）

平板**没有年龄概念**——分龄完全是服务器策略在同一套卡片词汇上的不同用法。这与服务器一期已实现的「按年龄段 profile、全局唯一生效」机制对齐：一期由家长在管理后台切 profile，二期增加 `set_profile` 推送让平板同步手势启用表与字数上限。

| 阶段 | 服务器策略典型输出 | 平板新增工作 |
|------|-------------------|-------------|
| 3-4 岁（v1 做扎实） | sketch 回应、stamp 奖励、count 数数、trace 描形状 | 无——v1 全集 |
| 5-6 岁（预留） | trace 换汉字 + 田字格、`page` 练习页、text 份量加重、「?」手势启用 | 零改动，纯服务器策略 |
| 6-7 岁（预留） | 笔顺评价（用回合上传的有序笔画+时间戳）、text 短文阅读 | 零改动 |

现在唯一要埋的钩子：**回合请求携带有序笔画与时间戳**（笔顺分析的原料）。设备已在采集（pen.rs 硬件事件率、memory.rs 有序存盘），缺的只是把每点时间戳保留下来并随回合上传。

## 10. 与手机端的配合

- 共享 `turn_id`：语音（手机播）与纸面卡片（平板画）属同一回合，两端各自尽快开始，**不做严格音画同步**——语音先响、笔迹跟上，自然时差恰似「一边说一边画」。
- 手机端可发起：语音提问回合（服务器附带平板当前页面）、换页、切活动（「豆豆，陪他玩描红吧」）、家长指令（退出/重启/睡眠）。
- 平板与服务器保持一条长连接（WebSocket 或长轮询，实现期定）；`commit_now` 等控制事件从这条连接推下。
- 手机不在/没网：平板退回纯停笔触发模式，语音缺席但纸面交互完整可用。
- **分期现实**（对齐服务器一期设计 §2）：riddle 目前只能主动发起请求、无法被推送，跨设备联动整体属于二期。一期的真实形态是「平板回合」与「手机语音回合」各自独立走同一个 Turn 引擎与同一个生效 profile——人格一致但不联动；家长陪坐时手机可对着平板页面拍照提问作为临时桥。

## 11. 电子纸表达边界

**适合**：线稿简笔画、逐笔动画、淡墨模板（描红）、大字、点数标记、留白多的排版。

**不做**：

- 灰度照片/位图渲染（唯一例外候选：远期故事插画页可考虑抖动灰度底图，见第 16 节）；
- 快速连续动画（残影 + 刷新闪烁）；
- 密集小字段落；
- 需要即时反馈的「游戏式」交互（即时感由语音通道补）；
- **彩色渲染**——Paper Pro 屏实为彩色（Canvas Color 3.0），但彩色波形刷新慢且闪烁，不适合高频回应。v1 全灰度；彩色留作远期低频「特别时刻」选项（如一幅作品完成时的彩色印章）。服务器 v1 不得下发依赖颜色语义的内容（「圈出红色的」属于语音+实物场景）；
- Markdown、表情符号、表格排版。

## 12. 离线与异常

- **纸永远能画**：服务器不可达时，平板是一块安静的写字板——笔迹、橡皮、本地换页、睡眠照常，只是没有回应。恢复连接后**不补发**积压回合（孩子早已画到别处，旧回应徒增困惑），从下一次触发正常开始。
- 服务器出错/超时（>30 秒）：思考墨点安静消失，不在纸上写错误信息（错误进日志与家长端）。孩子视角：豆豆这次没接话，仅此而已。
- 应用崩溃：takeover 的 systemd 停止钩子已保证 xochitl 恢复（现状保留）；重启后回到最近一页的存档状态。

## 13. 一期能力内交互 vs 二期新渲染能力

服务器一期平板零改动（riddle 继续 OpenAI 兼容协议）。据此把本设计拆成三层：

### 13.1 一期就能做的（纯服务器提示词/profile 工程，设备不动）

- **手写文字回复**是唯一渲染形式：短句、儿歌、古诗、大字。3-4 岁 profile 可以让回复退化为「一两个超短句 + 单个主题字」，由陪坐家长读出。
- **提示词级的准教学**：让模型用文字给分步画画建议（「先画一个大圆」）；回复中带一个大字供孩子照着描（黑墨、位置不可控，是描红的粗糙前身）。
- **召回与记忆**照常：门面保留了 riddle 记忆协议（`⟦show:N⟧`、`⁂` 转写），手写「给我看…」可用（需家长代写）。
- **手机语音独立可用**：完整语音闭环在手机页，但与平板不联动。
- 一期的诚实定位：**平板 = 会写字回你的魔法纸（家长陪读），手机 = 独立的语音伙伴**。3-4 岁孩子无法独立消费文字回复，一期体验依赖家长在场。

### 13.2 需要二期（/turn + riddle 改造）的

全部卡片语言（sketch/stamp/count/trace/page）、就地共画与布局器、双通道触发与 `commit_now`、回应不淡出、自动换页、控制通道、记忆上云、`set_profile` 手势档切换、有序笔画上传。

### 13.3 建议的「一期半」设备微改动（不依赖 /turn，各是一两行常量+重编译）

1. `IDLE_COMMIT` 2.8s → 6s（幼儿停顿容忍）。
2. 五指点按退出 → 五指长按 ≥3s（防手掌误退到原生系统）。
3. （可选）每点时间戳落盘，为笔顺数据提前积累。

这三项都是儿童安全/数据项，与协议无关，建议在孩子开始真实使用前完成。

## 14. /turn 契约需求清单（供后端设计引用）

一期（OpenAI 兼容接口、平板零改动）期间本节不生效；本节对应二期 /turn，**直接决定 /turn 的 schema**。以下是平板视角的完整需求面。

### 14.1 回合请求（平板 → 服务器）

```json
{
  "turn_id": "t-20260722-0831",
  "trigger": "pen_idle | voice_commit | page_full | recall_query",
  "page_png": "<base64 灰度整页>",
  "new_strokes": [
    [[0.31, 0.42, 0.55, 0], [0.32, 0.43, 0.61, 12], "…"]
  ],
  "page_state": { "page_id": "p-42", "ink_coverage": 0.63 },
  "device_profile": { "screen": [1620, 2160], "profile": "child_3_4", "app_version": "…" }
}
```

- `new_strokes`：本回合**新增**的有序笔画；每点 `[x_norm, y_norm, pressure_norm, t_ms]`，`t_ms` 为相对本回合首点的毫秒偏移。用途：笔顺分析（识字档）、增量理解（区分「刚画的」与「页面原有的」）。后端可先存不用。
- `ink_coverage`：0-1 页面满度，服务器判断「还发得下大卡片吗 / 该建议换页了吗」。
- `trigger=voice_commit` 时服务器负责把该回合与手机端语音文本合并成一个上下文。

### 14.2 回合响应（服务器 → 平板）

```json
{
  "v": 1,
  "turn_id": "t-20260722-0831",
  "spoken_text": "哇，三个太阳！我们一起数一数。",
  "paper_cards": [
    { "type": "stamp",  "name": "star", "count": 3,
      "place": "near_new_ink", "size": "S" },
    { "type": "count",  "n": 3, "style": "dots",
      "place": "near_anchor", "anchor_norm": [0.42, 0.31] },
    { "type": "sketch", "strokes": [[[0.10, 0.20], [0.15, 0.25]], "…"],
      "place": "blank_area", "size": "M", "pace": "slow" },
    { "type": "text",   "content": "太阳",
      "place": "near_anchor", "anchor_norm": [0.42, 0.31], "size": "L" },
    { "type": "trace",  "kind": "shape", "content": "circle", "guide": "none",
      "place": "blank_area", "size": "L" },
    { "type": "page",   "layout": [ { "card": { "type": "trace", "kind": "hanzi", "content": "山", "guide": "tian_grid" }, "rect_norm": [0.1, 0.1, 0.8, 0.3] } ] }
  ],
  "page_action": "none | suggest_new_page | new_page",
  "memory_tags": ["sun", "counting", "drawing"]
}
```

字段说明：

- 通用：`place`（`near_new_ink` / `near_anchor` / `blank_area` / `margin` / `full_page`）、`anchor_norm`（可选）、`size`（S/M/L）、`pace`（normal/slow）。
- `text.content`：短句/单字。
- `sketch.strokes`：归一化坐标（0-1，相对卡片自身包围盒）的有序折线组。
- `stamp.name + count`：名字必须来自平板公布的符号枚举，v1 共 8 个：`star / flower / heart / smiley / check / sun / moon / balloon`。
- `count.n + style`：`dots / tally / numbers`。
- `trace.kind + content + guide`：`shape`（`circle / square / triangle / star / heart / wave`）或 `hanzi`（单字）；`guide`：`none / tian_grid`（田字格）。
- `page.layout[]`：嵌套卡片 + 显式 `rect_norm [x, y, w, h]`——唯一允许服务器排坐标的地方（新白纸无竞态）。
- `spoken_text`：平板忽略，手机消费；列在此处是因为契约要求它与 `paper_cards` 语义一致。

### 14.3 平板强制约束（后端生成时必须遵守，平板会校验并截断）

1. 每回合 `paper_cards` ≤ 3 张；`page` 卡必须单独成回合。
2. `sketch` 折线总点数 ≤ 2000（保证逐笔动画约 10 秒内画完）。
3. `text.content` 长度按档位限制：`child_3_4` ≤ 6 字；识字档放宽（服务器经 `set_profile` 告知）。
4. 不得下发依赖颜色语义的内容；不得下发 Markdown/表情符号。
5. `spoken_text` 与 `paper_cards` 必须讲同一件事（语音解说纸面）。

### 14.4 控制通道（服务器 → 平板推送事件，长连接，二期）

| 事件 | 作用 | 典型发起方 |
|------|------|-----------|
| `commit_now` | 语音提问到达，平板立即封存当前页面成回合 | 手机语音 |
| `new_page` | 换页 | 家长/服务器 |
| `recall { page_id }` | 淡墨浮现某个存档页 | 语音「给我看…」 |
| `set_profile` | 切换儿童档/识字档（手势启用表、字数上限、空闲阈值） | 管理后台 |
| `parent_command { exit \| restart \| sleep }` | 家长遥控 | 手机 |

### 14.5 记忆同步

- 每页封存时平板上传完整存档：有序笔画（含时间戳）+ PNG + 回应卡片 + `memory_tags`。
- 服务器为记忆主库；平板本地目录为缓存。
- `recall` 响应需返回该页存档笔画数据，供平板淡墨逐笔重写。

## 15. 决策记录

| 问题 | 决策 |
|------|------|
| 首个目标用户 | 3-4 岁；语音（手机）为一等公民通道，纸面+语音同回合双落地 |
| 回合触发 | 双通道：停笔 6s（可配）+ 语音即时触发 |
| 回应形式 | 简笔画、奖励标记、认知标注全上；描红预留 |
| 笔顺采集 | 设备已有能力；回合请求带有序笔画+时间戳 |
| 回应位置 | 就地共画：孩子墨迹神圣不动，DouDou 画在旁边空白 |
| 页面结构 | 无模式单页流；活动由服务器驱动，家长语音切换 |
| 换页 | 自动（页满）+ 语音；板上无儿童换页手势 |
| 手势安全 | 五指改长按 ≥3s；「?」儿童档停用；橡皮/电源键保留；家长手机遥控 |
| 渲染分工 | 方案 A：服务器管内容与摆放意图，平板管几何与落笔 |
| 简笔画格式 | 服务器下发归一化有序折线；stamp 符号平板内置 |
| 屏幕色彩 | 屏为彩色（Canvas Color 3.0），v1 全灰度线稿，彩色留远期低频选项 |

## 16. 开放问题（留给实现期）

1. 长连接选型：WebSocket vs 长轮询（取决于后端窗口的技术栈与设备网络稳定性实测）。
2. 思考墨点的具体形态与位置（页角固定 vs 跟随最后落笔处），待真机手感调试。
3. `stamp` 符号库的美术定稿（8 个符号的矢量笔画序列）。
4. 页面「快满」阈值（`ink_coverage` 多少触发建议换页），待真实涂鸦数据调参。
5. 召回浮现在儿童档是否需要防误触保护（孩子乱说「给我看」高频触发打断当前涂鸦）——可由服务器策略限频。
6. `sketch` 卡内容的服务器侧来源，三条候选路线（按基建成本递增）：
   a. **素材库/QuickDraw 检索**：视觉模型认出孩子画的主题 → 从精选矢量涂鸦库（如 Google Quick, Draw! 数据集，345 类 5000 万笔画向量，CC BY 4.0）检索一幅干净笔画——零生成基建，永远渲染漂亮；
   b. **LLM 直接生成折线/SVG**：快、灵活，简单图形质量可接受；
   c. **扩散模型 sketch-to-image + 提线**：孩子涂鸦 PNG → ControlNet(scribble) 条件生成 → 线稿提取 → 骨架化成有序折线（riddle 的 Zhang-Suen 管线可搬到服务器复用）。Mac（Apple Silicon）上 1-5 秒/张，效果最丰富，基建最重。
   三条路线产出同一种 `sketch.strokes` 折线，契约不受影响，可并存按活动选用。
7. 远期故事插画页：抖动灰度位图底图（quill 已有 image 渲染 demo）是否引入为 `page` 卡的可选背景——与「不做位图」边界的取舍，留到故事课程设计时定。
